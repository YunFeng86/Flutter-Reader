## Context

仓库当前暴露出的不是单点 bug，而是边界表达失真：`Article.categoryId` 去规范化的收益判断在两份 benchmark 中采用了不同口径，Android 支持声明与 release signing 现状不匹配，README 对 Web 的结论、原因和保留命令彼此冲突，归档后的 OpenSpec 规格仍保留 `TBD` 占位文本。随着这些信号同时存在，后续贡献者很难判断哪些实现是正式承诺，哪些只是过渡性状态。

这次 change 关注的是“重建项目契约”，而不是立即推进 Android 发布链路产品化或直接移除 `Article.categoryId`。边界声明一旦统一，后续无论是继续保留去规范化、推进 Android 发布，还是显式冻结 Web，都能基于一致标准做决策。

## Goals / Non-Goals

**Goals:**
- 为平台支持状态建立统一、可验证的分层标准，避免“代码能编一点”就被表达成“已支持”。
- 为高维护成本优化建立统一的 benchmark 解释口径，让保留/复盘决策有清晰依据。
- 校正 README、benchmark 文案、发布配置说明与 OpenSpec 基线文档之间的漂移。
- 把仍未产品化的能力标记为预览或后续工作，减少仓库对外和对内的误导。

**Non-Goals:**
- 不在本次 change 中完成 Android 正式发布链路建设。
- 不在本次 change 中直接移除 `Article.categoryId` 或重构分类查询实现。
- 不把所有历史文档都重写一遍；只修正当前影响决策和认知的边界文档。

## Decisions

### 1. 采用三层边界模型，而不是“支持/不支持”二元模型

平台边界统一分为三层：
- 正式支持：已验证构建与运行，且具备基本发布准备度。
- 预览支持：代码路径存在，但仍缺少关键发布或验证条件。
- 未支持：当前构建或运行不可行，或明确不纳入当前产品承诺。

这样可以准确表达 Android 这类“代码存在但发布链路未完成”的状态，也能避免 Web 这类“当前构建失败却仍保留命令”的叙述继续混淆读者。

替代方案：
- 方案 A：维持现有“正式支持 / 理论支持 / 暂不支持”表述。
  放弃原因：这些标签目前没有统一准入条件，导致 README 与配置、构建结果之间可以长期脱钩。

### 2. 将 benchmark 从“证明实现正确”改为“支持保留复杂度决策”

`Article.categoryId` 相关 benchmark 的职责不应只是输出快慢，而应明确服务于“是否值得继续承担冗余字段与一致性维护成本”的决策。因此需要统一：
- 使用哪一种核心度量口径；
- 阈值如何表达；
- 低于阈值时是标记复盘、冻结新依赖，还是进入移除评估。

本次 change 不直接决定是否删除该字段，而是先把 benchmark 的解释契约统一下来，避免同一组数据在不同文件里得出不同结论。

替代方案：
- 方案 A：直接在本次 change 中决定保留或删除 `Article.categoryId`。
  放弃原因：当前更大的问题是判断标准漂移，先修正决策契约比直接拍板更稳。

### 3. 把 README 与 OpenSpec 一起视为“基线文档”，统一校正

这次 change 会同时覆盖 README、benchmark 文案和 OpenSpec 规格元数据，而不是只修 README。原因是仓库的边界认知不仅来自 README，也来自 benchmark 注释、归档 spec 的 Purpose、以及配置文件中的 release 注释。若只修一处，漂移会继续存在。

替代方案：
- 方案 A：只改 README，把其余问题留给未来慢慢清理。
  放弃原因：这会保留多套并存的事实来源，无法真正校正团队认知。

### 4. 优先校正文案和状态声明，只在必要时微调配置注释

本次 change 应优先更新事实声明与验收标准；只有当配置注释本身会误导支持边界时，才对配置注释或少量元信息做同步调整。这样能把范围控制在“边界校正”，避免顺着文档问题滑成 Android 产品化或数据模型重构。

替代方案：
- 方案 A：把 release signing、Web 兼容、去规范化重构全部并入同一 change。
  放弃原因：跨度过大，会把校正边界的工作变成多条产品/架构路线的并行实施。

## Risks / Trade-offs

- [边界校正后暴露出更多未完成事项] → 用“预览支持/后续工作”显式承接，而不是继续模糊表述。
- [统一 benchmark 口径后，当前实现看起来收益不足] → 将结果定义为触发复盘，而不是在本次 change 中仓促重构数据模型。
- [文档改动被误解为功能降级] → 在 proposal/README 中明确这是对真实状态的校正，不是新引入的限制。
- [OpenSpec 基线修正范围被持续扩大] → 仅处理当前 directly referenced 的占位与冲突项，其他文档债单独记录。

## Migration Plan

1. 盘点当前平台支持声明、构建命令、发布配置、benchmark 文案和 OpenSpec 基线中的冲突点。
2. 为 `project-boundary-contract` 建立规格，固定平台分层、benchmark 解释和文档一致性的验收边界。
3. 按规格更新 README、benchmark 注释/说明、相关配置注释和 OpenSpec 占位内容。
4. 将仍未满足正式支持条件的平台或仍未定案的优化标记为预览状态或后续工作。
5. 重新检查构建/文档/规格状态，确保同一事实在仓库内只剩一种表述。

回退策略：
- 若某项边界声明的事实依据短期仍不明确，保守降级为“预览支持”或“待验证”，避免继续过度承诺。

## Open Questions

- `Article.categoryId` benchmark 最终统一采用“节省时间百分比”还是“速度倍率”作为主度量，哪一种更符合团队后续阅读和决策习惯。
- README 是否需要把实验性运行命令单独移到“试验/开发验证”小节，而不是继续放在快速开始主路径里。
- Android 是否只需在文档中明确为预览支持，还是还需要同步补充一个最小 release readiness checklist 作为后续工作入口。
